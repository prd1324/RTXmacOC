# RTXmacOC — Архитектура и стратегия атаки

> Документ живой. Цель — зафиксировать **где стена и куда бить**, чтобы команда
> не распыляла силы. Обновляется по мере разведки.

## 1. Что мы на самом деле строим

Драйвер NVIDIA RTX (Ada, AD104, RTX 4070 Super, `10DE:2783`) для macOS,
работающей на **обычном x86-ПК под OpenCore (Hackintosh)**, где RTX стоит как
**основная видеокарта**.

**Это НЕ про:**
- eGPU/видеокарту, подключённую к настоящему Маку;
- Apple Silicon (другой boot, нет PCIe-слота под dGPU).

**Это про:** хакинтош-комьюнити (OpenCore, пользователи современных macOS вплоть
до Tahoe), у которых на macOS новее High Sierra **нет никакого драйвера NVIDIA** —
официальные NVIDIA Web Drivers умерли на 10.13. Название проекта: RTX + mac + **OC
(OpenCore)**.

«Драйвер» здесь — это не один бинарь, а стек из нескольких слоёв:

1. **PCIe bring-up** — найти карту, смапить регистры (BAR0), DMA.
2. **GPU init** — поднять чип до рабочего состояния.
3. **Memory management** — управление VRAM и адресным пространством GPU (GMMU).
4. **Command submission** — каналы/очереди команд в GPU.
5. **Display / modeset** — вывод изображения (интеграция с WindowServer).
6. **3D / compute accel** — бэкенд под Metal (самый верх, самый далёкий).

Месяц 1 целиком про слой 1 и разведку слоёв 2–6. Реалистично.

## 2. Ключевое открытие: Ada рулится через GSP

Начиная с **Turing**, NVIDIA перенесла низкоуровневую инициализацию GPU внутрь
самого чипа — на отдельный процессор **GSP (GPU System Processor)**, который
исполняет подписанную NVIDIA прошивку.

Источники (NVIDIA, публичные):
- Open-модули ядра «не поддерживают GPU до Turing, потому что зависят от GSP»
  ([NVIDIA README, kernel_open](https://download.nvidia.com/XFree86/Linux-x86_64/575.64/README/kernel_open.html)).
- «GSP firmware используется по умолчанию для всех Turing и более новых GPU»
  ([NVIDIA README, gsp](https://download.nvidia.com/XFree86/Linux-x86_64/565.77/README/gsp.html)).
- NVIDIA полностью открыла исходники ядра драйвера для Turing/Ampere/Ada/Hopper
  ([NVIDIA/open-gpu-kernel-modules](https://github.com/NVIDIA/open-gpu-kernel-modules)).

_Содержимое источников пересказано для соответствия лицензионным ограничениям._

### Почему это меняет всё

Раньше (эпоха nouveau) драйвер приходилось реверсить регистр за регистром —
годы работы. Для Ada **протокол общения с картой уже опубликован самой NVIDIA**
в виде open-gpu-kernel-modules. Значит:

- Слой 2 (GPU init) — это не слепой реверс, а **портирование известной логики
  RPC к GSP** на драйверную модель macOS.
- Наша основная сложность смещается с «как поднять карту» на **«как уложить это
  в модель драйверов Apple»**.

## 3. Где настоящая стена: модель драйверов macOS

Контекст наш особенный: **Hackintosh под OpenCore**. Это снимает часть барьеров,
которые есть у обычного Мака.

| Версия macOS | Модель | Что можно на Hackintosh |
|---|---|---|
| ≤ 10.13 High Sierra | kext + были офиц. NVIDIA Web Drivers | максимум доступа, но старо |
| 10.14–10.15 | kext (deprecated) | грузятся, SIP релаксится через OpenCore |
| 11+ (Big Sur → Tahoe) | kext (SIP relax) **или** DriverKit/PCIDriverKit (dext) | оба варианта реальны |

**Ключевое преимущество Hackintosh-контекста:**
- Пользователи OpenCore **штатно релаксят SIP** (`csr-active-config` в NVRAM) и
  **инжектят kext'ы** через OpenCore. Значит kext — не «запасной план», а **родной
  канал доставки**, как Lilu / WhateverGreen в их экосистеме.
- Мы встраиваемся в существующий тулинг OpenCore, а не пробиваем модель Apple в лоб.

Реальные ограничения DriverKit (если пойдём dext-путём), по Apple Dev Forums:
- Маппинг BAR — через PCIDriverKit; `MemoryRead*` работает, прямой доступ к BAR0
  крэшит при неверной настройке
  ([Apple DevForums 680945](https://developer.apple.com/forums/thread/680945)).
- DMA в память приложения — через дескрипторы страниц, без copy
  ([Apple DevForums 696778](https://developer.apple.com/forums/thread/696778)).

Вывод: и kext (через OpenCore), и dext умеют то, что нужно для bring-up. Начинаем
с того, что даёт больше контроля и привычно комьюнити.

## 4. Стратегия (Architecture Decision)

- **Платформа — x86 Hackintosh под OpenCore.** RTX как основная карта. Apple Silicon
  не наш кейс в принципе.
- **Канал доставки — kext, инжектируемый OpenCore** (наравне с dext как альтернатива).
  Это родной для комьюнити способ и даёт максимум доступа к железу.
- **Путь — портирование GSP-логики**, а не bring-up с нуля. open-gpu-kernel-modules
  как референс протокола общения с картой.
- **Метрика месяца 1** — драйвер/проба читает `PMC_BOOT_0` (offset 0x0 в BAR0) и
  выводит chip ID Ada. Доказательство, что канал к карте есть.

## 5. Риски (честно)

- Подписанная прошивка GSP: её надо где-то взять и загрузить — лицензионный и
  технический вопрос.
- Display-стек macOS (WindowServer/IOFramebuffer) закрыт — вывод картинки на
  Hackintosh нетривиален (комьюнити частично решает это для iGPU/AMD, для NVIDIA — нет).
- Metal-бэкенд закрыт полностью — самая дальняя и тяжёлая цель.
- Совместимость с цепочкой OpenCore (Lilu и пр.) и с будущими версиями macOS,
  которые могут урезать x86-поддержку.

Ни один из рисков не делает слой 1–2 невозможным. Их и берём первыми.
