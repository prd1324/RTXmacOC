# Графический стек macOS и блокер вывода изображения (Неделя 2)

> Открытый вопрос: **существует ли в современном macOS (Big Sur+) точка расширения
> для стороннего kext/dext как полноценного акселератора для WindowServer/
> IOGraphics, или эта модель закрыта Apple?**
>
> Короткий ответ: **публичной/поддерживаемой точки расширения НЕТ. Путь к
> ускоренному выводу через WindowServer закрыт не технически, а политикой Apple
> (code signing + приватные интерфейсы + DriverKit без графики).**

Это блокер уровня всего проекта, не «потом». Ниже — что именно закрыто и чем
подтверждается.

---

## Как macOS рисует на экране (упрощённо)

1. **WindowServer** — процесс Apple, который композит экрана.
2. Для ускорения он грузит **Metal GPU driver bundle** — userspace-библиотеку
   (dylib) — **в своё адресное пространство**.
3. В ядре GPU обслуживают `IOGraphics`/`IOFramebuffer` (вывод) и `IOAccelerator`
   (ускорение), к которым привязан Metal-бандл.

Историческая модель (до 10.13): вендор поставлял kext (IOAccelerator) + Metal
bundle. Так работали NVIDIA Web Drivers. Этот путь **умер**.

---

## Три замка, каждый из которых фатален

### Замок 1 — Library Validation (с 10.14) блокирует Metal-бандл
Metal/OpenGL-ускорение требует загрузки GPU-бандла вендора в адресное
пространство процесса (WindowServer и приложений). С macOS 10.14 Apple включила
флаг **library-validation** практически везде: в процесс можно грузить только
библиотеки, подписанные **тем же разработчиком, что и сам процесс, или Apple**.
Значит сторонний GPU-бандл **не загрузится в WindowServer** — «dead in the water».
Источник (тех. разбор): [HN 20249176](https://news.ycombinator.com/item?id=20249176).

### Замок 2 — Нет публичного API для GPU-драйверов
Apple **не открывала** API написания GPU-драйверов. Поэтому нет драйверов даже
для виртуальных GPU в VMware/Parallels/VirtualBox/QEMU — композитор на Metal на
них не работает. Источник: [HN 21333408](https://news.ycombinator.com/item?id=21333408).

### Замок 3 — DriverKit не поддерживает графику; kext-путь сворачивается
- **DriverKit/dext не предназначен для графических/дисплейных драйверов** —
  GPU/дисплей остаются в ядре и у Apple. (Apple Dev Forums по DriverKit; графики
  среди поддерживаемых классов нет.)
- **kext'ы deprecated** с Big Sur, KPI урезаются; направление Apple — убрать
  любой сторонний код из контекста процессов Apple (kext→dext, hardened runtime,
  library validation). Источник: [Apple DevForums 651183](https://developer.apple.com/forums/thread/651183),
  [WWDC2019 session 702 (пересказ)](https://developer.apple.com/forums/thread/130224).
- IOAccelerator/IOFramebuffer-интерфейсы для Metal — приватные, недокументированные,
  только Apple, и меняются между версиями.

_Содержимое источников пересказано для соответствия лицензии._

## Вывод для RTXmacOC

| Цель | Достижимо сторонним драйвером на Big Sur+? |
|---|---|
| Поговорить с картой (PCIe, BAR0, регистры) — слой 1 | да (kext) |
| Поднять GPU/GSP, память, каналы — слои 2–4 | технически да (kext, headless/compute), без вывода |
| **Ускоренный вывод через WindowServer (Metal) — слой 5–6** | **нет** — Замки 1–3 |
| Базовый неускоренный фреймбуфер для произвольной PCIe-карты | практически нет на современном macOS; WindowServer завязан на Metal |

**Главное:** даже если мы доведём слои 1–4 до идеала (GSP жив, RPC работает,
карта инициализирована), **штатно отдать картинку в WindowServer мы не можем** —
это упирается в подпись и приватные интерфейсы Apple, а не в наш код.

## Что это меняет (без выбора решения — только фиксация)

Прежняя формулировка цели проекта («полноценный вывод изображения на RTX в
актуальной macOS сторонним драйвером») **противоречит модели Apple Big Sur+**.
Это надо признать до того, как писать ещё код слоёв 2+.

Возможные направления (НЕ выбраны, требуют отдельного решения с владельцем
проекта) — каждое со своими «но»:
- **Compute-only** (CUDA-подобный счёт без вывода) — обходит WindowServer, но это
  не «драйвер RTX для Mac» в исходном смысле.
- **Целевая ОС ≤ 10.13 High Sierra**, где сторонний GPU-стек ещё разрешён — но это
  не «актуальные macOS».
- **Патч/обход library validation и приватных интерфейсов** — нестабильно,
  ломается каждым апдейтом, юридически серая зона, фактически не «поддержка».

Решение по направлению — за пределами этого документа. Документ фиксирует **факт
блокера**.
